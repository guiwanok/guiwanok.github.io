ReentrantLock 源码分析

发表于 2018-03-30 | | 阅读次数:

简介

网络上已经有很多ReentrantLock的源码分析，写这篇文章是因为看了很多文章，还是有很多困惑的地方，所以在自己分析完源码后，想写一篇文章记录一下分析过程，对以前有困惑的地方做一些记录。

首先ReentrantLock有公平锁和非公平锁两种实现方式，其实现方式非常相似。这里先介绍一下非公平锁。公平锁也是基于非公平锁的实现上做了改造，保证持有锁的线程，释放锁时唤醒且保证竞争锁成功的一定是等待最久的线程。这里表述上有点复杂，等下面具体分析再看。

成功加入队列后，还需要对线程进行阻塞操作，acquireQueued方法会完成阻塞操作，除了阻塞外，还有其他操作哦。

首先先来了解一下Node结点状态这个字段：

waitStatus 含义
1 表示当前线程已被取消
-1 表示当前节点的后置结点包含的线程需要运行
-2 表示当前节点在等待condition，也就是在condition队列中
-3 表示当前场景下后续的acquireShared能够得以执行
0 表示当前节点在sync队列中，等待着获取锁

waitStatus=-1 的含义非常容易混淆，因为它并非表示当前结点的状态，而表示的是后置结点的状态。

acquireQueued方法无限循环体里，前半部分很好理解，判断当前结点的前置结点是否是head，如果是表示当前结点就是队首，尝试获取锁，如若成功，则直接返回。可以看出，先进入队列的线程，最有可能最先被唤醒，但是在竞争的过程中，并不保证一定能获取到锁，这也是公平锁和非公平锁的差别，公平锁的实现后面会讲。

后半部分，进入shouldParkAfterFailedAcquire()方法，这个方法判断前置结点的状态，若为-1则返回true，另外还维护了队列，排除了状态大于0的结点（因为这些结点的线程已经被取消）。当前置结点的状态为0或者-3时，则CAS修改前置结点的状态为-1，并返回false，重新进入到无限循环体重。这里笔者也很困惑，已经将前置结点改为-1了，下一次循环会进入首个判断，并返回true。为什么还需要再一次循环呢？笔者判断还是为了尽量避免线程进入阻塞状态，多一次尝试多一点获取锁的可能，这点还未考证过，只是个人猜想。

shouldParkAfterFailedAcquire()返回true后，进入parkAndCheckInterrupt()方法，该方法最终调用UNSAFE.park()试线程进入阻塞状态，这是一个native方法。

至此ReentrantLock的加锁流程已分析完毕。

公平锁与非公平锁的区别

来看看具体实现有什么差异

最重要的区别是公平锁在尝试加锁时，一定会判断是否有前缀正在等待获取锁的结点。如果有，则放弃获取，转入等待队列中。也就是说想要获取锁，必须满足结点是头部结点的后一个结点，也就是第一个等待获取锁的结点。由于aqs的实现是将获取锁失败的线程，总是加入到等待队列的队尾，所以可知队首的结点代表的线程一定是等待最久的线程。

实现上，获取到锁的线程释放锁后，唤醒的也是排在队首的线程，区别在于尝试获取锁的操作没有做限制，当且仅当仅队首线程可以获取到锁。也就是说当释放锁时，还没来得及唤醒队首线程，可能已经被其他线程占得先机，拿到锁，这种情况有点类似于插队。
