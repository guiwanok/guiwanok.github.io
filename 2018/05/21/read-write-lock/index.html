ReentrantReadWriteLock 源码分析

发表于 2018-05-21 | 分类于 java | | 阅读次数:

简介

ReentrantReadWriteLock是ReadWriteLock读写锁的实现类，维护了两把锁：读锁和写锁。读锁是共享锁，多个线程可以同时获取读锁，相互不冲突。写锁是排它锁，同一时刻仅限一个线程可以获取写锁，实现逻辑同ReentrantLock。同时，读锁和写锁相互冲突，即当线程A持有读锁，线程B想要获取写锁时，线程B会被阻塞，被加入到同步队列中。

结构分析

ReentrantReadWriteLock提供读锁ReadLock和写锁WriteLock，均为内部实现类。读锁和写锁内部持有的sync为同一个对象，即等待队列为共享的，sync的实现分为NonfairSync和FairSync，分别表示非公平锁和公平锁，其区别在于是否严格按照FIFO方式处理唤醒同步队列中的等待线程。这一点跟ReentrantLock的实现类似。

先来看看aqs.acquiredShared()做了什么：

非常简单的逻辑，先尝试获取读锁，如果获取成功，直接返回，如果获取失败，则进入doAcuquireShared方法，加入同步队列并阻塞当前线程等操作，一会再具体分析。先来看看关键步骤，tryAcquireShared方法，其实现类sync到底在这个方法中实现了怎样的逻辑。

第一次看到这么大段代码是会有点懵，这里大致解释一下执行逻辑：

  1. 如果已有线程持有写锁，返回失败
  2. 检查同步队列中是否有被写锁阻塞的线程
  3. 检查读锁个数是否超限
  4. 如果2/3成功则执行cas更新读锁状态state
  5. 如果4成功，则修改reader和holderCount等属性
  6. 如果2/3/4有一部失败，则进入循环重试

这里需要细化一下，首先介绍一下 ReentrantReadWriteLock是怎么保存读写锁的state。由于cas只能操作单个属性，所以读写锁是公用一个state。state是一个int类型，ReentrantReadWriteLock将state一分为二，前16位表示读锁状态，后16位表示写锁状态。所以读锁和写锁的最大数量是不能超过2^16 - 1个。

另外firstReader和firstReaderHoldCount是做什么的呢？

前面提到过读锁是共享锁，可以被多个线程公共持有，firstReader记录了当前第一个持有读锁的线程，而firstReaderHoldCount则记录了firstReader持有读锁的数量。当并发冲突时，即多个线程同时获取独读锁时，通过ThreadLocalHoldCounter来记录每个线程持有读锁的数量。ThreadLocalHoldCounter继承自ThreadLocal，能够和线程进行绑定，存储在线程私有空间内，天然解决了并发安全问题。

这里笔者第一次看的时候有些疑惑，已经记录了读锁状态state，为什么还需要记录每个线程的HoldCount？试想读锁和写锁一样，加锁和解锁的步骤都必须由同一个线程执行，如果只记录读锁状态state，那么是不是线程A加锁之后，其他线程也可以操作解锁了呢？读锁因为是共享锁，可以被多个线程持有，自然需要记录每个线程持有读锁的次数了。写锁其实也要求加锁和解锁必须由同一个线程完成，只是因为写锁是排它锁，同一时刻仅限一个线程持有，那么只需要在lock内开辟一块空间记录这个thread即可。这么一想，就能理解为什么要记录HoldCount了呢。

那么这里又衍生出另外一个问题，为什么只记录firstReader和firstReaderHoldCount，其他线程又需要ThreadLocalHoldCounter来记录读锁次数，为什么不全部使用ThreadLocalHoldCounter呢？

这里笔者猜测是处于性能的考虑，比如并发度不高的情况下，其计数操作会尽量落到firstReaderHoldCount上，不会涉及threadLocal的操作，相比threadLocal的操作，性能方面还是有所提升。另一个点，lock本身有一个属性cachedHoldCounter，用于记录上一次操作的HoldCounter对象，可以避免再次操作threadLocal.get方法，也是出于性能优化的考虑。

同样是一个模板方法，tryReleaseShared方法由子类实现。

逻辑如下：

  1. holdCounter减一
  2. 读锁state减一
  3. 当读锁state等于0表示，读锁完全解除，返回true，否则返回false

如果cas执行失败则自旋重试，同样的套路。

doReleaseShared方法会从同步队列中取出首结点，并唤醒首结点的线程。

至此读锁的加锁和解锁过程已经介绍完毕。做一个小的总结，读锁维护了一个状态state，记录读锁的加锁次数，加锁时state++，当前线程的加锁次数++。解锁时需要将这两个状态–，并判断state是否等于0，如果等于0，则唤醒同步队列首节点线程。
